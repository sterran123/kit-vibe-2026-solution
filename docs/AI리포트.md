## 1. 기획

### ① 실제 타 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

**타겟 사용자 3그룹**

| 사용자 | 구체적인 문제점 |
|--------|----------------|
| **수강생** | 학습 중 이해 안 되는 부분이 생겨도 즉각적인 피드백을 받을 수 없어 학습 흐름이 끊김. 강사에게 질문을 남겨도 응답까지 기다려야 하며, 어떤 맥락에서 질문했는지 설명하는 것 자체가 번거로움. AI 채팅/강사 질문 시스템의 UI/UX 일관성 부족으로 학습 흐름이 단절되고, 알림 및 질문 관리 시스템이 사용자 친화적이지 않아 학습 정보 확인이 불편함 |
| **강사(교강사)** | 수강생의 반복적인 기초 질문 처리로 본업(강의 준비, 심화 피드백)에 집중하기 어려움. 학생이 남긴 질문만으로는 어떤 맥락에서 막혔는지 파악하기 어려워 답변 준비에 추가 시간이 소요됨 |
| **교육 운영자(관리자)** | 강사 등록 시 자격증/강사증 진위 여부를 수동으로 검토해야 하는 비효율. 부정 등록 강사 차단을 위한 검증 프로세스가 체계화되어 있지 않음 |

---

### ② 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

**TutorBridge의 7가지 핵심 기능**

**1. AI 튜터 (24/7 즉각 응답)**
- Gemini 2.5 Flash 기반 스트리밍 응답으로 응답 지연 없이 대화
- 이미지, PDF, 코드 파일 첨부 분석 지원 (Gemini Vision 활용)
- 대화 세션을 Firestore에 저장해 학습 히스토리 유지
- 대화창 자동 생성 및 대시보드와 실시간 동기화

**2. Contextual Highlight Menu**
- AI 답변 중 이해 안 되는 텍스트를 드래그(2자 이상)하면 팝업 메뉴 등장
- "추가 설명", "강사에게 질문" 등 즉시 액션 가능 — 맥락이 자동으로 포함됨

**3. Smart 티켓 에스컬레이션 (사이드 패널)**
- AI가 해결하지 못한 문제를 강사에게 에스컬레이션할 때, AI가 대화 맥락을 자동 요약해서 첨부
- 수강 중인 강사 목록에서 질문 대상 강사를 직접 선택 — 강사별 질문 분류
- 강사는 처음부터 전체 대화를 읽지 않아도 핵심 상황을 즉시 파악 가능
- 티켓 목록에 제목(질문 번호)이 아닌 질문 내용 미리보기 표시

**4. 강사 답변 조회 (대시보드 통합)**
- 대시보드에서 내 질문 티켓과 강사 답변을 한눈에 확인
- 강사 답변 미리보기, 상태(대기/진행중/해결됨), 담당 강사명 표시
- 답변 알림 클릭 시 해당 티켓 사이드 패널로 바로 이동

**5. 통합 알림 시스템**
- 읽음 처리 및 개별 삭제 버튼으로 알림 관리 최적화
- 전체 읽음 처리로 알림 쌓임 방지
- 답변 알림 클릭 시 해당 티켓으로 바로 연결

**6. AI 자격증/강사증 자동 검증**
- 강사 신청 시 업로드한 자격증 이미지를 Gemini Vision이 분석
- 이름·생년월일 일치 여부, 공식 기관 발급 여부, 위조 가능성 등을 수치화하여 관리자에게 제공

**7. 실시간 강사 대시보드 & 관리자 패널**
- Firestore onSnapshot 기반 실시간 티켓 수신 및 상태 관리
- 관리자: 강사 승인/거절 + 이메일 알림 발송 (Firebase Functions)

### ③ 이 솔루션이 도입되었을 때 기대되는 개선 효과는 무엇인가요?

- **수강생**: AI 튜터가 기초·반복 질문을 즉시 해소 → 학습 연속성 유지. 정말 어려운 문제만 강사에게 에스컬레이션되므로 강사 응답 대기 시간 단축. 대시보드에서 강사 답변을 즉시 확인 가능 → 학습 정보 단절 해소
- **강사**: AI가 기초 질문을 필터링 → 강사는 심화 피드백에 집중 가능. 티켓에 AI 요약이 첨부되어 맥락 파악 시간 대폭 단축. 수강생이 직접 강사를 지정하므로 질문 분류 효율 향상
- **관리자**: 자격증 검증 자동화로 검토 시간 단축 및 부정 등록 차단 체계화

## 2. AI 활용 전략

### 이 프로젝트에서 사용한 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

| 구분 | 도구/모델 | 용도 |
|------|-----------|------|
| **개발 보조** | Windsurf (Cascade) | 아키텍처 설계, 전체 코드 구현 |
| **개발 보조** | Claude Code (claude-sonnet-4-6) | 코드 리뷰, 리팩토링, UI/UX 개선, 버그 수정 |
| **프로덕트 AI** | Gemini 2.5 Flash | AI 튜터 응답, 파일 분석, 자격증 검증 |

**선택 이유**

- **Windsurf**: `.windsurf/rules/set.md`에 프로젝트 아키텍처 규칙과 기술 스택을 명문화해두고, Cascade가 이를 자율적으로 참고하며 일관된 코드를 생성하도록 활용
- **Claude Code**: 기존 구현에 대한 독립적인 코드 리뷰와 레거시 코드(Cloudflare Workers, Express 서버) 정리 목적으로 도입. Windsurf가 구현한 코드를 별도 관점에서 검증하고, `/simplify` 등 내장 스킬을 통해 코드 품질을 지속적으로 개선
- **Gemini 2.5 Flash**: 무료 티어(15 RPM)로 공모전 데모 운영이 가능하며, Google Search 그라운딩·멀티모달(이미지·PDF) 지원이 교육용 AI 튜터에 최적. 스트리밍 SSE 지원으로 응답 지연 체감 최소화

---

### 각 AI 도구별 활용 전략

#### Windsurf (Cascade) — 주력 개발 AI

**핵심 전략: Rules-as-Architecture**

`.windsurf/rules/set.md`에 15년 차 수석 아키텍트 페르소나와 함께 아래 규칙을 명문화:
- `public/` 폴더 내 HTML/CSS/JS 철저한 파일 분리 (단일 파일 집중 금지)
- AI API 직접 호출 금지 → 반드시 서버 경유 (Firebase Functions)
- CDN 방식 라이브러리 우선 (npm 의존성 최소화)
- **부분 패치(Partial Patch)**: 변경된 함수/클래스 단위만 수정 (전체 재작성 금지)

이 규칙이 있어야 AI가 매 요청마다 일관된 아키텍처 원칙을 유지하고, 불필요한 토큰 낭비 없이 타겟 코드만 수정한다.

**단계별 활용 흐름**

```
Phase 1 → HLD 작성 (시스템 아키텍처, 기술 스택, ERD)
Phase 2 → LLD 작성 (API 명세, DB 스키마)
Phase 3 → 구현 (HTML/JS/CSS 컴포넌트 단위)
Phase 4 → Firebase Functions 구현 (Gemini API 연동, 자격증 검증)
```

**에이전트 구성 방식**

Cascade를 단일 에이전트로 운용하되, `set.md` 규칙이 상시 로드(`trigger: always_on`)되어 모든 요청에 컨텍스트가 유지됨. 별도 멀티 에이전트 구성 없이 rules 파일 하나로 일관성 확보.

#### Claude Code (claude-sonnet-4-6) — 코드 리뷰 & 개선 AI

**핵심 전략: 독립적 2차 검토 + 지속적 품질 개선**

Windsurf가 구현한 코드를 Claude Code가 독립적으로 리뷰하여, 같은 AI가 자기 코드를 검토하는 편향을 방지.

**이번 작업에서 수행한 내용**:

1. **레거시 정리**: 미사용 파일 식별 및 제거 (`server.js`, `worker/`, `1234.html`). Cloudflare Workers 이메일 엔드포인트가 실제로 미구현 상태임을 발견 → Firebase Functions + nodemailer로 이전
2. **UI/UX 개선 시리즈**:
   - 전체 6개 페이지 모바일 반응형 적용 (notification 드롭다운 `position:fixed`, 하단 탭바)
   - `classroom.html` 신규 구현 (Alpine.js + ES Module 로딩 레이스 컨디션 해결)
   - 강사 대시보드 2×2 통계 그리드, 프로필 수정 모달, 테마/언어 전역 적용
3. **코드 품질 개선 (`/simplify` 스킬 활용)**:
   - `instructorsWithCourses` getter O(n²) → 단일 pass reduce로 개선
   - `loadStudents()` N+1 쿼리 → `Promise.all()` 병렬 실행
   - `onmouseover` 인라인 핸들러 → CSS `.dropdown-item` 클래스로 교체
   - `logout(await _fbDash)` Alpine 인라인 await 오류 → `doLogout()` 메서드 분리
4. **기능 개선**:
   - 알림 개별 삭제 + 전체 읽음 처리
   - 티켓 발행 시 수강 강사 선택 드롭다운 (사이드 패널 방식)
   - 티켓 목록 제목: 티켓 번호 → 질문 내용 미리보기
   - 대시보드에 강사 답변 조회 카드 추가
   - 드래그 최소 글자 수 10자 → 2자로 완화

#### Gemini 2.5 Flash — 프로덕트 AI (Firebase Functions 경유)

**구현 구조**

```
브라우저 → Firebase Functions (인증/sanitize) → Gemini 2.5 Flash API
```

클라이언트가 Gemini API를 직접 호출하지 않고 Firebase Functions를 경유함으로써:
- API 키 노출 방지
- Firebase Auth 토큰 검증 후 요청 처리
- Prompt Injection 필터링 (20개 패턴)

**멀티모달 활용**
- 텍스트 파일/코드: base64 디코딩 후 텍스트로 전달
- 이미지/PDF: `inlineData` 방식으로 Gemini Vision에 직접 전달
- 자격증 검증: 이름·생년월일·발급기관·위조 여부를 JSON으로 응답받아 점수화

### 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략

**1. Partial Patch 강제 (Windsurf Rules)**

`set.md`에 "변경된 함수/클래스 단위의 부분 패치 방식만 사용, 전체 재작성 금지"를 명시. AI가 수정이 없는 파일을 재탐색하거나 전체 파일을 다시 작성하는 토큰 낭비를 구조적으로 차단.

**2. CDN 라이브러리 = 빌드 단계 제거**

Tailwind CSS, Alpine.js, Firebase SDK 모두 CDN 방식 적용. `npm run build`가 없으므로 번들 분석·최적화 관련 AI 대화 자체가 불필요해짐.

**3. Gemini 2.5 Flash 선택**

Gemini 2.5 Pro 대비 응답 속도가 빠르고 비용이 낮으며, 무료 티어(15 RPM/분, 1500 RPD/일)가 공모전 데모 운영에 충분. 기능 차이가 튜터링 품질에 영향을 주지 않는 수준임을 테스트로 확인.

**4. Firestore 세션 캐싱**

대화 히스토리를 Firestore `chat_sessions` 컬렉션에 저장하고, Gemini 요청 시 전체 히스토리를 재전송하지 않고 세션 ID로 조회해 필요한 부분만 포함. 대화가 길어져도 Gemini에 전달하는 컨텍스트 크기를 제어 가능.

**5. 점진적 업데이트 전략**

UI 개선 사항을 단위별(알림 기능 → 티켓 시스템 → 채팅 동기화 → 질문 관리 순)로 작업하여 버그 추적 용이성 확보. Claude Code의 `/compact` 명령으로 컨텍스트 압축 후 이어서 작업하는 방식으로 긴 개발 세션에서도 일관성 유지.

**6. Circuit Breaker 원칙**

동일 에러 해결을 3회 이상 시도해도 실패하면 코드 수정을 멈추고 가설을 보고하는 원칙을 `set.md`에 명시. 토큰 폭주를 구조적으로 방지.
